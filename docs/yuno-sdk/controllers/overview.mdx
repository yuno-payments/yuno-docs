title: "Overview"

Controllers are how your code receives the SDK's events and, when needed, overrides default behavior. Implement only the ones you need.

## Defaults

| Controller | Not overridden | Overridden |
|---|---|---|---|---|
| [`onStatus`](/docs/yuno-sdk/controllers/on-status) | You don't receive the final status. | Receives the final transaction status (`status`, `substatus`, `message`) once the transaction reaches a final state. |
| [`showView`](/docs/yuno-sdk/controllers/show-view) (iOS, Android) | SDK presents forms and action views (e.g. 3DS) full-screen. | SDK hands you the view and a `TransactionProcess`; embed it and call `process.onSubmit()`. |
| [`showLoading`](/docs/yuno-sdk/controllers/show-loading) | SDK shows its own full-screen loader. | SDK suppresses its loader and emits `isLoading: true/false`. |
| [`createPayment`](/docs/yuno-sdk/controllers/create-payment) | SDK creates the payment after tokenization. | SDK delivers the One-Time Token; you create the payment from your backend and resolve. |

Implement `onStatus` in every integration. The others are optional.

## How to wire controllers

- **Web.** Each event is a plain function set directly as fields on the transaction config (`onStatus`, `showLoading`, `createPayment`).
- **iOS, Android.** Implement `TransactionController` on the object that should react to events (your view model, activity, or a dedicated class) and pass it through the `controller` field of `TransactionConfig`. Each event maps to a method on that object. `showView` is available on mobile only.

## Threading

On iOS and Android, every controller method is invoked on the main thread. You can update UI directly from inside it. Move any blocking work (e.g. backend calls inside `createPayment`) off the main thread yourself.
