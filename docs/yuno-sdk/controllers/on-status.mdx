---
title: "onStatus"
description: "Receives the final status of the transaction."
---

`onStatus` is invoked when the transaction reaches its final state. It is the only way to know how the transaction ended.

## Behavior

This controller has no default behavior. If you don't implement it, you won't receive the final status. When implemented, the SDK delivers the final `PaymentStatus` to your handler after tokenization, payment creation (or merchant-side payment creation via `createPayment`), 3DS, and status resolution have completed.

## Signature

<Tabs>
  <Tab title="Web">

Wire it as a field on the config.

```typescript
sdkPayments.transaction({
  ...,
  onStatus: (result) => console.log(result.status),
})
```

`result` is `PaymentResult` for payment flows and `EnrollmentResult` for enrollment. Both wrap `status` (the literal union from v1's `Status` / `EnrollmentStatus`) with `substatus` and, for enrollment, `vaultedToken` when applicable.

  </Tab>

  <Tab title="iOS">

```swift
func onStatus(status: SDKPayment.PaymentStatus)
```

  </Tab>

  <Tab title="Android">

```kotlin
fun onStatus(status: PaymentStatus)
```

  </Tab>
</Tabs>

See [`Status`](/docs/yuno-sdk/reference/status) for the shape of the payload, the `message` field, and the SDK-emitted values (`INTERNAL_ERROR`, `USER_CANCELLED`).

## Threading

On iOS and Android, `onStatus` is invoked on the main thread. You can update UI directly from inside this handler.
