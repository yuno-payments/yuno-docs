description: "Receives the final status of the transaction."

`onStatus` is invoked when the transaction reaches its final state. It is the only way to know how the transaction ended.

## Behavior

This controller has no default behavior. If you don't implement it, you won't receive the final status. When implemented, the SDK delivers the final status to your handler after tokenization, payment creation (or merchant-side payment creation via `createPayment`), 3DS, and status resolution have completed.

## Signature

<Tabs>
  <Tab title="Web">

Wire it as a field on the config.

```typescript
sdkPayments.transaction({
  ...,
  onStatus: (result) => console.log(result.status),
})
```

`result` carries `status` and `substatus` (backend values passed through) plus `message` on error outcomes — the same payload shape on payment and enrollment flows. There is no separate result type per flow.

  </Tab>

  <Tab title="iOS">

```swift
func onStatus(status: SdkPayments.TransactionStatus)
```

`SdkPayments.TransactionStatus` carries `status` and `substatus` (raw backend values, passed through verbatim) and an optional `message` populated on error outcomes.

  </Tab>

  <Tab title="Android">

```kotlin
fun onStatus(status: TransactionStatus)
```

  </Tab>
</Tabs>

See [`Status`](/docs/yuno-sdk/reference/status) for the shape of the payload, the `message` field, and the SDK-emitted values (`INTERNAL_ERROR`, `USER_CANCEL`).

## Confirm on your server

<Warning>
`onStatus` is a client-side signal. Use it to drive your UI — never as the source of truth for fulfillment.
</Warning>

`onStatus` runs on the user's device, so it can be missed (the app is killed, the network drops, the user closes the page mid-flow) or manipulated (anything running on the client can be tampered with). A `SUCCEEDED` delivered to your handler is not proof that money moved.

Before shipping goods, granting access, or marking an order as paid, confirm the outcome on your backend:

- **Webhooks (recommended).** Configure [webhooks](/docs/webhooks) so Yuno notifies your backend of every payment status change, independently of what happens on the client. Verify the notification signature before trusting it.
- **Query the API.** Have your backend call [Retrieve Payment by ID](/reference/payments/retrieve-payment-by-id) (or [by `merchant_order_id`](/reference/payments/retrieve-payment-by-merchant-order-id)) and read the status from the response.

Treat `onStatus` as the trigger to update the screen, and the server-side confirmation as the trigger to fulfill.

## Threading

On iOS and Android, `onStatus` is invoked on the main thread. You can update UI directly from inside this handler.
