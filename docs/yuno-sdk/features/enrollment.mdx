Enrollment lets you tokenize and save a customer's payment method without processing a charge. Use it to register a card or other method that the customer can use in future payments.

The same `Transaction` class handles enrollment. Only the session type changes.

## Before you start

Your backend must create the resources required for enrollment:

- Create the customer. See [`POST /v1/customers`](https://docs.y.uno/reference/customers/the-customer-object).
- Create the customer session. See [`POST /v1/customers/sessions`](https://docs.y.uno/reference/customer-sessions-enrollment/create-customer-session).
- Create the payment method. See [`POST /v1/customers/sessions/{customer_session}/payment-methods`](https://docs.y.uno/reference/payment-methods-checkout/enroll-payment-method-checkout).

## Implementation

Pass the `customerSession` (instead of `checkoutSession`) in `TransactionConfig` and call `start(paymentSelected:)` with the method the customer chose, the same way you do for a payment. The SDK delivers the final status to `onStatus` once enrollment is complete.

Rendering and selecting the payment method to enroll is your responsibility. The SDK does not present a list of enrollable methods. To fetch the available methods, call [`GET /v1/customers/sessions/{customer_session}/payment-methods`](https://docs.y.uno/reference/payment-methods-checkout/retrieve-payment-methods-to-enroll-checkout).

### Skip the form

If you collect the customer's data in your own UI, pass it via `transactionData` (`data` on Web) on the `PaymentMethodSelected`. The SDK will skip its built-in form and tokenize directly — a headless flow, so `elementSelector` is not needed. See [`TransactionData`](/docs/yuno-sdk/reference/transaction-data) for the full data shape.

<Note>
`vaultedToken` on `PaymentMethodSelected` is ignored during enrollment. The flow's purpose is to produce a new vaulted token, so there's nothing linked yet to reference.
</Note>

## Notes

- A 3DS challenge can occur during enrollment.
- `createPayment`, if overridden, is not invoked with payment data. There is no payment to create.
- When enrollment succeeds, `onStatus` delivers the final status — the same payload shape as payment flows (`status`, `substatus`, `message`). The vaulted token is not surfaced to the client: retrieve it from your backend's list-customer-payment-methods endpoint after the success status arrives, and store it against your customer to use in future payments.
