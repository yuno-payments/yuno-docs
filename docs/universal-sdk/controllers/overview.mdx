Controllers are how your code receives the SDK's events and, when needed, overrides default behavior. Implement only the ones you need.

## Defaults

| Controller | Not overridden | Overridden |
|---|---|---|
| [`onStatus`](/docs/universal-sdk/controllers/on-status) | You don't receive the final status. | Receives `PaymentStatus` once the transaction reaches a final state. |
| [`showView`](/docs/universal-sdk/controllers/show-view) | SDK presents forms and action views (e.g. 3DS) full-screen. | SDK hands you the view and a `TransactionProcess`; embed it and call `process.onSubmit()`. |
| [`showLoading`](/docs/universal-sdk/controllers/show-loading) | SDK shows its own full-screen loader. | SDK suppresses its loader and emits `isLoading: true/false`. |
| [`createPayment`](/docs/universal-sdk/controllers/create-payment) | SDK creates the payment after tokenization. | SDK delivers the One-Time Token; you create the payment from your backend and resolve. |

Implement `onStatus` in every integration. The others are optional.

## How to wire controllers

- **Mobile (iOS, Android).** Implement `TransactionController` on the object that should react to events (your view model, activity, or a dedicated class) and pass it through the `controller` field of `TransactionConfig`. Each event maps to a method on that object.
- **Web.** Each event is a plain function. Either set the functions directly as fields on the transaction config (`onStatus`, `showView`, `showLoading`, `createPayment`), or chain them on the `Transaction` instance with `transaction.on('status' | 'view' | 'loading', fn)` and `transaction.createPayment(fn)`.

If you both pass a function in the config and chain one for the same event on web, the chained function wins.

## Threading

On iOS and Android, every controller method is invoked on the main thread. You can update UI directly from inside it. Move any blocking work (e.g. backend calls inside `createPayment`) off the main thread yourself.
