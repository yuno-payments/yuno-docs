# Style Guide for SDK Files

This guide defines how to structure and write SDK documentation across Web, Android, and iOS. It synthesizes the Technical Writing - Good Practices Guide and approved feedback from the SDK cleanup review.

## 1. Language and tone
- Use American English spelling, grammar, and terminology.
- Tone: professional, clear, instructional, and concise.

## 2. Document structure
- Title hierarchy: H1 Page title; H2 sections (Overview/Steps/Complementary features); H3 for steps or subtopics; H4 sparingly.
- Keep the original section order unless there is a strong reason to change it.

## 3. Required sections and conventions
- Step 1: Include the library in your project.
  - Always show explicit inclusion for the platform (script tag, npm, gradle, CocoaPods, pubspec, etc.).
- Step 2: Initialize SDK with the public key.
  - Use code identifiers: `PUBLIC_API_KEY` (never bolded text for identifiers).
  - Credentials phrasing: “See the credentials page for more information.” Link “credentials” to the appropriate page.
- Start checkout / Mount SDK steps:
  - Prefer phrasing that avoids repeating function names in prose when they are immediately shown in a code block.
  - Example: “Display the payment methods by using:” then show the `yuno.mountCheckout()` block.

## 4. Parameters and duplication
- Use a single titled parameters table per method.
  - Table title should be “Parameters” before the table to avoid ambiguity.
  - Move inline code comments that explain parameters into the table.
  - Add a short introductory sentence before the table (one line) to set context (e.g., “Configure X with the following options:”).
- Deduplicate: if parameters appear in multiple places, centralize in one Parameters section and refer back to it.

### Canonical parameter descriptions
- `language`: Use this exact description everywhere (Web/Android/iOS):
  - "| `language`    | Defines the language to be used in the payment forms. You can set it to one of the available language options: <ul><li>es (Spanish)</li><li>en (English)</li><li>pt (Portuguese)</li><li>fil (Filipino)</li><li>id (Indonesian)</li><li>ms (Malay)</li><li>th (Thai)</li><li>zh-TW (Chinese (Traditional, Taiwan))</li><li>zh-CN (Chinese (Simplified, China))</li><li>vi (Vietnamese)</li><li>fr (French)</li><li>pl (Polish)</li><li>it (Italian)</li><li>de (German)</li><li>ru (Russian)</li><li>tr (Turkish)</li><li>nl (Dutch)</li><li>sv (Swedish)</li><li>ko (Korean)</li><li>ja (Japanese)</li></ul>    |"
  - Keep the list synced with `web-sdk-integrations/sdk-v12/full-sdk-v12.md`.

## 5. Terminology and naming
- “Render mode” (not “mode of form rendering”).
- “Issuer’s form” (not “form of the issuer”).
- Method reference: `ContinuePayment` when used as a method name in headings/callouts; `continuePayment` in code.
- Headings: prefer “Example” or “Usage” instead of “Example usage”.

## 6. Code blocks and commentary
- Keep explanations above or below code blocks, not inline within the code.
- Remove redundant or verbose inline comments in code samples.
- Ensure syntax highlighting is correct; keep code runnable and minimal.
- Avoid multi‑line explanatory comments inside code blocks. Move multi‑line notes, caveats, or parameter explanations into callouts (tips/warnings) or tables for better readability and skimmability.

## 7. Callouts and optional features
- Place optional/advanced notices at the start of their section.
- Replace unclear callouts with clear, actionable guidance on the same page.
- Avoid sending readers to an unrelated page to complete core steps; keep instructions self‑contained.

### Required callouts
- Credentials callout: after the first initialization example on a page, add a callout that links to credentials.

Example:

> 📘 Credentials
>
> See the credentials page for more information: https://docs.y.uno/reference/authentication

## 8. Linking and navigation
- Use descriptive link text (e.g., “See the credentials page…”).
- Verify all internal anchors and references (Parameters section anchors, complementary features anchors, etc.).

## 9. Consistency rules (cross‑SDK)
- Align terms, headings, and phrasing across Web, Android, and iOS.
- Use the same step names across variants where applicable.
- Use “render mode,” “issuer’s form,” and `ContinuePayment` consistently.

## 10. Per‑page checklist (apply to each SDK doc)
- [ ] American English and tone compliance.
- [ ] Clear title and section hierarchy (H1/H2/H3).
- [ ] Step 1 shows explicit library inclusion for the platform.
- [ ] Step 2 uses `PUBLIC_API_KEY` and “See the credentials” phrasing.
- [ ] Headings use “Example” or “Usage,” not “Example usage”.
- [ ] Parameters consolidated in a titled “Parameters” section; no duplication.
- [ ] Explanations outside code; code free of redundant inline comments.
- [ ] Terminology: “render mode,” “issuer’s form,” `ContinuePayment` method.
- [ ] `mountCheckout` phrasing avoids repetition in prose.
- [ ] Optional/advanced notes at the start of their sections.
- [ ] Internal links and anchors validated; code blocks correctly formatted.

## 11. Web SDK specifics (examples)
- Render mode section:
  - Title: “Render mode”.
  - If showing the object structure, ensure the `renderMode` example is complete and valid.
- Continue Payment:
  - Use a callout titled “`ContinuePayment` Method” that explains when `sdk_action_required` is true and shows where to call `yuno.continuePayment()`.

## 12. Non‑goals
- Do not alter API behaviors or introduce new flows in docs.
- Do not expand scope beyond the SDK page’s intent; cross‑link instead.
